Aufgabe 1.1 - Fachliche Argumentation über Erfolgsprinzipien des WWW

Mit welchen fachlichen Argumenten wurde das WWW-Proposal von TBL abgelehnt?

Man hielt den Ansatz von TBL für einen Rückschritt, da durch seine Vorschläge "broken-links" entstehen konnten. Dieses Problem meinte man bereits gelöst zu haben.

Was sind die fachlichen Argumente, warum das WWW dennoch ein Erfolg wurde?

Das Konzept von TBL sah es vor, dass das Protokoll, die Auszeichnungssprache und die Identifikation von Ressourcen losgelöst voneinander entwickelt werden konnten. Der Erfolg gibt TBL recht. Auch heute, ca. 30 Jahre später, wird immer noch auf seiner Basis weiterentwickelt.

Was wäre der Preis für die garantierte Verhinderung von “broken links”?

Zur Vermeidung von "broken-links" müsste ein zentrales Verzeichnis angelegt werden, in dem alle Ressourcen verwaltet werden. Neue Webseiten müssten sich bei diesem Verzeichnis anmelden und abgeschaffte Seiten müssten abgemeldet werden. Lösungen zur Löschweitergabe müssten implementiert werden, damit die referenzielle Integrität gewahrt werden kann.

Aufgabe 1.2 - HTTP

Sie bekommen im Browser den HTTP Status Code 200. Was bedeutet das?

Die Anfrage ist erfolgreich verlaufen und wird mit dem Status-Code 200 Ok bestätigt.

Sie bekommen im Browser den HTTP Status Code 301. Was hat das zu bedeuten?

Die angefragte Ressource wurde dauerhaft verschoben. Es wird ein redirect zur neuen Quelle mittels des Location-Feldes im Header durchgeführt.

Sie bekommen im Browser den HTTP Status Code 400. Was hat das zu bedeuten? Was können Sie dagegen tun?

Hierbei handelt es sich um einen Bad Request. Dies bedeutet, dass der Server die Anfrage nicht bearbeiten kann oder vielleicht auch nicht bearbeiten will. Probleme könnten fehlende rechte sein, die durch einen Log-in erlangt werden können. In den meisten Fällen ist jedoch die Ressource oder Anfrage nicht ausführbar. Man könnte also die Anfrage mit der bereitgestellten API abgleichen.

Sie bekommen im Browser den HTTP Status Code 403. Was hat das zu bedeuten? Was können Sie dagegen tun?

Hierbei handelt es sich um einen Forbidden Response. Dies bedeutet, dass der Server die Anfrage zwar bearbeiten könnte, dies aber aufgrund fehlender Berechtigungen verhindert. Ein Log-in bzw. eine Erneuerung der Session könnte Abhilfe schaffen.

In einer Webanwendung benötigen Sie eine OPTIONS-Anfrage, die die Optionen des Servers vor dem eigentlichen Zugriff erfragen soll. Aus Performanzgründen soll die Abfrage jedoch cacheable sein. Wie könnten Sie dafür eine Lösung angehen?

Optionsabfragen sind idempotent. Das bedeutet, dass eine Anfrage beliebig oft gestellt werden kann und immer das gleiche Ergebnis liefert. Da das Ergebnis immer gleich ist, kann auch einfach nach einmaliger Abfrage der Rückgabewert gespeichert werden.
